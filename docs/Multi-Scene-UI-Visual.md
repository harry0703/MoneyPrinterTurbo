# 多场景UI改进方案

## 1. 方案概述

本方案旨在改进多场景模式下的UI界面，提供更直观、易用的用户体验。采用统一的多场景架构，将过去的无场景模式映射为单场景的多场景模式，多场景模式永远激活，不再需要模式切换。

## 2. 核心设计

### 2.1 界面选择

**选择方案**：卡片式界面

**理由**：
- 适合场景数量较多的情况
- 可以快速浏览所有场景
- 空间利用更充分
- 在移动设备上也有较好的显示效果

### 2.2 卡片结构

每个场景卡片包含以下元素：

| 元素 | 描述 |
|------|------|
| 场景信息 | 场景序号、预计时长、实际音频时长 |
| 视觉需求 | 详细描述输入框 + 关键词标签（如 #城市 #夜景 #蓝色调） |
| 视频关键词 | 输入框，用于指定该场景的视频素材搜索关键词 |
| 文案编辑 | 多行文本框，支持基本格式化 |
| 操作按钮 | 预览、编辑、删除、复制、上移/下移 |

### 2.3 交互功能

- **场景管理**：添加、编辑、删除、复制场景
- **顺序调整**：通过上移/下移按钮调整场景顺序
- **批量操作**：全选、批量删除、批量复制
- **保存机制**：自动保存编辑内容，防止意外丢失

```python
# 初始化状态

    st.session_state["video_terms"] = ""
```

## 4. 技术实现

### 4.1 状态管理

- 使用 `st.session_state` 集中管理所有场景数据
- 采用结构化数据存储（JSON数组）
- 实现状态变更的原子操作

### 4.2 性能优化

- 使用 `st.container` 和 `st.empty` 实现局部更新
- 避免在每次交互时重新渲染整个页面
- 采用懒加载策略，只渲染当前可见的场景
- 将耗时操作移至后台线程

### 4.3 兼容性

- **向后兼容**：将现有无场景模式自动映射为单场景的多场景格式
- **数据转换**：实现无场景数据到多场景数据格式的自动转换
- **平滑迁移**：确保现有项目数据在新架构下正常工作

## 5. 视觉设计

### 5.1 卡片样式

- 轻微阴影、圆角边框、hover效果
- 不同场景类型使用不同颜色标识
- 显示场景完成度进度指示
- 空状态时显示引导信息

### 5.2 响应式设计

- **桌面端**：多列布局，充分利用屏幕空间
- **平板端**：双列布局
- **移动端**：单列布局，优化触摸操作

## 6. 辅助功能

- **模板库**：提供常用场景模板
- **文案建议**：基于视觉需求的AI文案推荐
- **时长估算**：实时估算音频时长
- **静态预览**：编辑时实时预览文案布局和视觉效果

## 7. 错误处理

- 添加全面的错误捕获机制
- 实现状态恢复功能，在出错时保持UI稳定
- 提供清晰的错误提示，避免静默失败

## 8. 实现步骤

1. **统一架构实现**：移除无场景模式，统一使用多场景架构
2. **数据迁移**：编写脚本将现有无场景项目转换为单场景多场景格式
3. **多场景UI实现**：开发卡片式界面，支持场景管理
4. **文案设置UI实现**：直接显示所有文案相关的输入和操作按钮
5. **功能完善**：添加场景管理、顺序调整等功能
6. **性能优化**：实现局部更新和懒加载
7. **测试验证**：确保统一架构的稳定性和兼容性

## 9. 预期效果

- **用户体验**：统一、直观的多场景管理界面，无需模式切换
- **性能**：流畅的操作体验，避免UI断联
- **兼容性**：现有无场景项目自动迁移到多场景架构
- **扩展性**：为未来功能扩展预留空间，统一架构简化开发

## 10. 风险评估

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| UI断联 | 影响用户体验 | 优化状态管理，实现局部更新 |
| 性能问题 | 处理大量场景时卡顿 | 实现懒加载和虚拟滚动 |
| 数据丢失 | 意外刷新导致数据丢失 | 实现自动保存和状态恢复 |
| 兼容性问题 | 数据迁移过程中的格式错误 | 编写严格的数据转换脚本，提供回滚机制 |
| 用户适应 | 用户需要适应新界面 | 提供清晰的引导和帮助信息 |

## 11. 结论

本方案通过统一的多场景架构设计，将过去的无场景模式映射为单场景的多场景模式，消除了模式切换的复杂性。采用卡片式界面设计，结合直接的文案设置UI，为用户提供了直观、高效的多场景管理界面。

这种统一架构不仅简化了代码结构，也为用户提供了一致的操作体验，同时为未来功能扩展预留了更大的空间。通过数据迁移和兼容性保障，确保了现有项目的平滑过渡。这将显著提升用户在处理视频文案时的体验和效率，无论是单场景还是多场景的需求都能得到满足。
